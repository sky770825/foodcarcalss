# 前後端對照檢查

## 資料表 `foodcarcalss` 欄位對應

| 資料庫欄位 | 前端 (script.js) | 後台 (admin.js) | 說明 |
|------------|------------------|-----------------|------|
| id | id, rowNumber | rowNumber / id | 主鍵，前後端一致 |
| vendor | formData.vendor / event.title | row.vendor, editVendor | ✓ |
| food_type | formData.foodType, foodType | row.food_type, editFoodType | ✓ |
| location | formData.location (來自 location 選單) | row.location, editLocation | 前端送的是 locationConfigs 的 key（如 漢堡大亨、四維路59號） |
| booking_date | formatDateForDisplay(formData.date) → "X月Y日(星期Z)" | row.booking_date, formatDateForDisplay(editDate) | 寫入格式一致；後台篩選/顯示皆用同一格式 ✓ |
| status | '己排'（固定） | row.status, formatStatusDisplay → "已排班" | ✓ |
| fee | formData.fee \|\| '600元/天' | row.fee \|\| '600元/天' | ✓ |
| payment | 新預約寫入 '尚未付款'；接手/釋出時依邏輯 | row.payment；正規化為 未繳款/己繳款/逾繳可排 | 後台將「尚未付款」「未付款」皆視為未繳款 ✓ |
| note | '' | row.note, editNote | ✓ |
| payment_image_url | 上傳時帶入；update 時寫回 | row.payment_image_url，新預約卡片/列表可顯示 | ✓ |
| created_at | - | 用於新預約 24h 判斷、排序 | ✓ |

## 付款狀態 (payment) 對照

| 情境 | 前端寫入/顯示 | 後台辨識 | 備註 |
|------|----------------|----------|------|
| 新預約 | '尚未付款' | 未繳款（與 未繳款、未付款 同組） | ✓ |
| 已付款 | 顯示「✓ 已付款」；更新時送 '己繳款' | 己繳款、已付款 → 已付款篩選 | ✓ |
| 逾期可排 | 顯示 event.payment | 逾繳可排；自動更新邏輯（>24h 未繳→逾繳可排） | ✓ |

## 場地 (location) 對照

- **前端**：`locationConfigs` 的 key 為選單值（四維路59號、四維路60號、漢堡大亨、自由風、蔬蒔、金正好吃），送出時即為該 key。
- **後台**：`locationNameMap` 與 `getLocationKeyForDisplayName()` 可對應「四維路70號」↔「漢堡大亨」等，篩選與編輯下拉一致。
- **唯一約束**：`(location, booking_date, vendor)`，前後端皆以同一組欄位寫入/更新，無衝突。

## 日期格式 (booking_date)

- **寫入**：前端與後台皆用 `formatDateForDisplay()` → `"X月Y日(星期Z)"`（如 2月5日(星期三)）。
- **讀取**：前端、後台、統計、逾繳判斷皆能解析該格式或 ISO；月份篩選有 `isBookingInMonth()` 等一致邏輯。✓

## 重複預約防護

- **前端**：新預約使用 `upsert(..., { onConflict: 'location,booking_date,vendor' })`，若已存在則更新。
- **後台**：僅對 `foodcarcalss` 做 update/delete（依 id），不 insert；編輯時若改為與他筆重複，會觸發 DB 23505，後台已顯示「此場地+日期+餐車已有預約，請勿重複或更改為其他組合」。

## 其他對齊項目

- **狀態顯示**：後台將「己排」顯示為「已排班」→ `formatStatusDisplay()`。
- **錯誤處理**：前端對 unique/ON CONFLICT 有專用提示；後台對 23505、401、403、42501 有對應訊息。
- **匯款圖片**：前端上傳寫入 `payment_image_url`；後台讀取並顯示於新預約卡片與列表；前端讀取預約列表時未映射 `payment_image_url`（目前前端不顯示匯款圖，僅後台顯示，故可接受）。

---

**結論**：前後端在欄位、付款狀態、場地、日期格式與重複防護上已對照一致，無發現遺漏或衝突。
